Routing Logic

Routing is determined by Case Type + Priority. The Team/Queue field captures the destination. The alert method captures the urgency.

Bug/Defect · Integration Issue · Performance/ReliabilityMedium / LowDev queue · Tracey monitorsQueue / async

Feature Request / EnhancementAnyTracey logs → Joe + Jose at product cadenceProduct backlog / cadence meeting
How To · Config/Setup · Process/OperationalAnyTracey ownsNo alert needed
Access / Permissions · Data IssueAnyTracey owns · Dev assist if platform-sideDev only if platform access required

Rule of Thumb
Dev gets tagged on anything requiring a code fix or platform-side investigation. Alec gets looped when it is client-facing severity — Critical, service disruption, compliance, or any escalated case past its expected resolution date. Joe and Jose are the destination for Feature Requests only — not active triage participants.

Section 5

Awaiting Response — Rename Recommendation

The Status dropdown already includes Waiting on Customer, which captures the state where client action is needed. The Awaiting Response checkbox as currently named describes the same condition — making it redundant.

The recommendation is to rename the Awaiting Response checkbox to "Awaiting Internal Response" — signaling that the ball is in Dev or Alec's court, not the client's. This is a field rename only, no logic change required.

Status: Waiting on Customer

  • Progress is blocked on the client
  • A follow-up has been sent and a client response is required
  • Already exists — no change needed
  • Unset when: client responds → back to In Progress

Checkbox: Awaiting Internal Response (renamed)

  • Progress is blocked on an internal teammate — Dev or Alec
  • A question or request has been sent internally and a response is needed
  • Field rename only — no logic change required
  • Unset when: internal teammate responds, work resumes
Usage Rule
A case can be both Waiting on Customer (status) and Awaiting Internal Response (checkbox) at the same time — waiting for client confirmation AND waiting for Dev to validate a fix. They are independent signals, not alternatives.
Section 6

Escalation Field Best Practices

The three escalation fields — Status = Escalated, the Escalated checkbox, and Escalated To — are a compound action. All three fire together or none of them do.

Hard Rule
A case should never have the Escalated checkbox checked without Escalated To populated and Status set to Escalated. Any partial state creates ambiguity about whether the case is truly escalated and being actively managed.

The Escalation Checklist

  • Set Status = Escalated
  • Check the Escalated checkbox
  • Populate Escalated To with a specific teammate — never a team or queue
  • Paste the Transform Ticket Link if a Dev ticket has been created
  • Add Escalation Notes: what is blocked, what the escalated person needs to do, urgency level
  • Notify the escalated person directly via Slack — do not rely on the platform to alert them

When to Escalate

  • Any Critical or High case not resolved within the working target
  • Any case requiring a code fix or platform-side investigation (→ Dev)
  • Any case involving data integrity, security, or compliance (→ Alec)
  • Any case where the client has expressed urgency beyond normal channels
  • Any case not progressing after 5+ business days with no clear resolution path
  • Any Incident or Service Disruption affecting multiple clients

De-escalation Rules

  • When Dev or Alec resolves their piece, update Status back to In Progress
  • Uncheck the Escalated checkbox
  • Update Escalation Notes with what was resolved and by whom
  • Reassign to the appropriate owner to complete resolution
  • Do not close a case while the Escalated checkbox is still checked — de-escalate first
Section 7

Operational Best Practices

Triage

TDaily triage habit

Build a morning check of all New cases. Any case in New status overnight without being triaged is a risk. Target: all New cases triaged within 24 hours of submission.

5The 5-field minimum

Account, Priority, Type, Assignee, and Expected Resolution Date must all be set before a case leaves Triaged status. If any are missing, the case is reviewed — not triaged.

→Triage is not a holding state

A case should not remain in Triaged status for more than 1 business day. If it does, the assigned owner has not acknowledged or begun work.

!Critical bypass

Critical and High cases bypass normal queue order — alert Dev and/or Alec immediately, then complete triage fields. Speed of first response takes priority over perfect classification.

Resolution Notes

RRequired to close

Resolution Notes must be populated before any case moves to Resolved or Closed. This is a standing operational rule — not yet enforced by platform logic.

KWhat a good note includes

(1) Root cause. (2) What was done to fix it. (3) Client communication or verification step. Over time these become searchable answers for recurring issues.
Good vs. Bad Resolution Note
Bad: "Fixed." or "Resolved by Dev." — No root cause, no reuse value.

Good: "User lacked the correct role in Transform. Added CX Viewer role to their account. Walked client through verification on call 3/15. Root cause: roles not provisioned during initial onboarding. Added to onboarding checklist."

Transform Ticket Link — How to Use It

When a case is escalated to Dev and a ticket is created in the Dev backlog, paste the ticket URL into the Transform Ticket Link field. This gives you a direct line of sight into:

  • Whether Dev has reviewed the issue
  • Whether it is in design, in progress, or in a sprint
  • Which sprint it is assigned to and what the expected delivery window is
  • Whether it has been deprioritized or is blocked internally

Check the linked ticket as part of the daily escalation review. If a ticket has not moved in 5+ days, that is the signal to follow up with Dev directly — not to wait for them to update the case.

Feature Requests must be logged in the product backlog with Jose before closing the case. Closing the case does not mean the request was heard.

Log: client name, specific use case, business impact, and how many accounts have made the same request. Volume signals priority to product.

Section 8

Triage vs. Resolution

TriageResolution
PurposeClassify, prioritize, and route the caseInvestigate, fix, communicate, and close the case
OwnerAlways Tracey — triage belongs to CX regardless of where the case routes nextAssigned owner (CX, Dev, or both)
Duration5–15 minutes per caseHours to days depending on complexity
What happensAccount, Priority, Type, Assignee, Expected Resolution Date set. Alerts sent if Critical/High. Transform Ticket Link added if Dev ticket exists.Work performed. Client updated. Notes captured throughout. Resolution Notes required before close.
Status resultNew → Triaged → In Progress (within 1 business day)In Progress → Resolved → Closed
Failure modeTriage becomes a holding state — cases sit indefinitely with no active owner and no expected resolution dateCase closes without Resolution Notes, leaving no institutional memory
Key Distinction
Triage answers: What is this, who owns it, and how urgent is it?
Resolution answers: What was wrong, what fixed it, and is the client satisfied?

The lifecycle runs left column top to bottom. Three branch states can occur mid-lifecycle on the right. Cancelled is a terminal state reachable from any point before Resolved.

Closed vs. Cancelled
Closed means the case reached a resolution — work was done, the client was satisfied, Resolution Notes exist.

Cancelled means the case was withdrawn before resolution — no fix was needed or delivered. Using Closed for unworked cases pollutes resolution metrics.
